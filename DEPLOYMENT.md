# Deployment checklist (Phase 13)

This is a working checklist for the first production deploy, not a
guarantee — go through it in order and check off each section.

## 1. Vercel project

1. Push this branch to GitHub (remote is already `John-jdj/AutomateWithJohn`).
2. In the Vercel dashboard: **Add New Project** → import the GitHub repo.
   Framework preset should auto-detect as Next.js.
3. Project Settings → General → Node.js Version: use **20.x or newer**
   (Next.js 16 requires it).
4. Leave the build command as `next build` (default) — `postinstall`
   already runs `prisma generate` automatically on every install, so the
   Prisma Client regenerates on every Vercel build without extra config.

## 2. Environment variables

Set these in Vercel → Project Settings → Environment Variables. Use
`.env.example` as the source of truth for names/format. Everything below
is grouped by what breaks if it's missing.

### Required for the app to function at all
- `DATABASE_URL`, `DIRECT_URL` — Supabase connection strings
- `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,
  `SUPABASE_SERVICE_ROLE_KEY`
- `NEXT_PUBLIC_SITE_URL` — set to the real production URL (used in auth
  redirect links, email links, JSON-LD, sitemap/robots). **Do not leave
  this as `http://localhost:3000` in production.**

### Guarded integrations (app works without these, just no-ops)
Every one of these follows the same pattern: unset in production means
that feature silently degrades rather than fails. Set the ones you're
ready to turn on:
- `RESEND_API_KEY`, `RESEND_FROM_EMAIL` — transactional email (Phase 10).
  Without it, password reset falls back to Supabase's own email sender;
  everything else (welcome/contact/invoice/newsletter emails) just skips.
- `NEXT_PUBLIC_RECAPTCHA_SITE_KEY`, `RECAPTCHA_SECRET_KEY` — spam
  protection on contact/consultation/comment forms.
- `OPENAI_API_KEY` — chatbot (Phase 11). Without it, the widget shows a
  static "chat isn't set up yet" message and never calls OpenAI.
- `NEXT_PUBLIC_GA4_MEASUREMENT_ID` — Google Analytics 4 (below).
- `NEXT_PUBLIC_CLARITY_PROJECT_ID` — Microsoft Clarity (below).
- `NEXT_PUBLIC_GOOGLE_SITE_VERIFICATION` — Search Console (below).

### Auth providers (configured in Supabase, not Vercel)
`GOOGLE_CLIENT_ID`/`SECRET` and `GITHUB_CLIENT_ID`/`SECRET` are listed in
`.env.example` but aren't actually read by this codebase directly —
they're configured in the Supabase dashboard under Authentication →
Providers, using `https://[project-ref].supabase.co/auth/v1/callback` as
the OAuth callback URL on the Google/GitHub side. Do that in Supabase, not
Vercel env vars.

## 3. Supabase production checklist

- Authentication → URL Configuration: set **Site URL** and **Redirect
  URLs** to the real production domain (the app currently builds
  `${NEXT_PUBLIC_SITE_URL}/auth/callback` for email confirmation, OAuth,
  and password reset links — Supabase will reject redirects to URLs not
  on this allowlist).
- Confirm Row Level Security policies migrated correctly — check
  `prisma/migrations/20260719140000_client_rls_policy` applied cleanly
  (RLS is enabled by default on Supabase tables with **zero** policies
  unless a migration explicitly adds them; this has bitten earlier
  phases).
- Run `npx prisma migrate deploy` against the production database before
  first traffic. **Never** `prisma migrate dev` against production (see
  the comment at the top of `prisma/schema.prisma` for why — it diffs
  against a shadow DB that doesn't have Supabase's `auth` schema).

## 4. Google Analytics 4

1. https://analytics.google.com → Admin → create a property → add a Web
   data stream for the production domain.
2. Copy the **Measurement ID** (`G-XXXXXXX`) into
   `NEXT_PUBLIC_GA4_MEASUREMENT_ID`.
3. `components/analytics/GoogleAnalytics.tsx` picks it up automatically
   on the next deploy — no code changes needed.

## 5. Google Search Console

1. https://search.google.com/search-console → Add property → the
   production domain.
2. Choose the **HTML tag** verification method (not DNS — simpler here
   since the app already supports it). Copy just the `content="..."`
   token into `NEXT_PUBLIC_GOOGLE_SITE_VERIFICATION`.
3. After deploying, click Verify in Search Console.
4. Submit the sitemap: `https://[your-domain]/sitemap.xml` (already
   generated by `app/sitemap.ts`).

## 6. Microsoft Clarity

1. https://clarity.microsoft.com → New project → production domain.
2. Copy the **Project ID** into `NEXT_PUBLIC_CLARITY_PROJECT_ID`.
3. Picked up automatically by `components/analytics/Clarity.tsx`.

## 7. Post-deploy smoke test

Once deployed with real env vars, manually walk every user-facing flow
once (this is the "final smoke test" BUILD_PLAN.md Phase 13 calls for —
it can only be done for real against the live deployment, not locally):

- [ ] Sign up with email, confirm the email arrives and the link works
- [ ] Log in / log out; OAuth (Google, GitHub) if configured
- [ ] Submit the contact form, consultation form, and a blog comment —
      confirm notification + confirmation emails arrive
- [ ] Subscribe to the newsletter — confirm the confirmation email
- [ ] Open the chatbot widget, ask a question, confirm it answers from
      real service data and that lead capture creates a Lead in
      `/admin/leads`
- [ ] As an admin: create an invoice, mark it SENT, confirm the client
      gets the invoice email
- [ ] Check GA4 Realtime report shows the visit; check Clarity records a
      session recording
- [ ] Verify security headers are present: `curl -I https://[domain]`
      should show `Content-Security-Policy`, `X-Frame-Options`, etc.
      (Phase 12)
- [ ] Spot-check a couple of rate limits didn't accidentally trip in
      normal usage (Phase 12) — `/admin/leads` should not show duplicate
      leads from retried form submissions
